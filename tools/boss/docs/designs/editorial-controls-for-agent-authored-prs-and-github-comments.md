# Boss: Editorial Controls for Agent-Authored PRs and GitHub Comments

## Problem

A Boss worker writes free-form text into several GitHub-visible surfaces
during a normal run: the pushed branch name, the PR title, the PR body,
review-comment replies, issue / PR comments it creates with `gh`, and
the commit-message body that ships with the PR. Today the only thing
shaping that text is whatever lives in `CLAUDE.md` / `AGENTS.md` plus
the model's own taste. That's fine for the author's personal repos.
It is **not** fine for repos where workers run alongside humans:

- **Internal taxonomy leaks.** The worker's prompt is dense with
  `exec_18b07a506d2518d0_1b`, `chg_8c0185712eef4cb7ba213279d8cf4146`,
  `proj_18a2bb…`, `task_18a2…`, "Boss worker", "cube lease",
  "the engine spawned you", phase numbers, work-item ids. The model
  routinely echoes these into PR titles, descriptions, and comments
  because they read as legitimate context. They are not — they are a
  private vocabulary the wider org has no reason to see, and at work
  they read as "this PR was machine-generated by a tool we did not
  authorise."
- **Branch names reflect the worker's origin.** The engine sets the
  push target to `boss/{execution_id}` by default; the branch prefix
  is already configurable per product. The execution id component is
  useful — it lets the engine reliably associate a PR branch with its
  originating execution. In repos where the `boss/exec_…` prefix is
  unwanted, a custom prefix can be set; see the branch-naming strategy
  below.
- **Repo PR-template conventions are ignored.** Most work repos ship
  `.github/PULL_REQUEST_TEMPLATE.md` (sometimes
  `.github/PULL_REQUEST_TEMPLATE/<kind>.md` for a multi-template repo)
  and humans fill those in. An agent's freeform "Summary / Validation /
  Open Questions" PR body sticks out like a sore thumb against twenty
  human PRs that all share the same template skeleton.
- **Social conventions vary by repo.** "No merge-conflict resolution
  comment on PRs," "no AI co-author trailer in commit messages,"
  "no emoji in PR titles," "PR body must reference a Jira ticket" —
  these are enforced socially in each environment. An agent that
  doesn't know them creates friction that's expensive to clean up
  retroactively (force-push, edit-PR, apologise in Slack).

The product needs a first-class place to attach **editorial rules** to
a Boss product (the natural scope, since one product owns one repo's
worth of work) and have those rules show up everywhere the agent
produces GitHub-visible text. The approach is undecided — prompt
injection, output sanitiser, gh-proxy sub-agent, hook-side enforcement,
or a combination. This document picks one.

---

## Goals

- A per-product **editorial-rules** record that the user (not the
  worker) controls, persisted in the engine's DB rather than in the
  repo, so rules can mention sensitive patterns without checking them
  into a repo with broader read access.
- Rules apply to every GitHub-visible surface a Boss worker writes to:
  branch name, PR title, PR body, commit-message body, PR / issue
  comments, review replies, merge-conflict-resolution notes.
- A **sane default** for *every* product (including personal ones)
  that strips known Boss-internal identifier shapes from worker
  output, without the user having to write anything. The opt-in
  ergonomic is "I want repo-template conformance"; the opt-out
  ergonomic is "strip identifiers from PR text but leave my
  branch-naming and template defaults alone."
- Two complementary enforcement points:
  1. **Up-front guidance**: editorial rules are injected into the
     worker's initial prompt so the worker writes compliant text on
     the first try.
  2. **Last-line-of-defence enforcement**: a deterministic hook
     intercepts `gh pr|issue {create,edit,comment,review}` invocations
     and either rewrites the body in place (for redactable taxonomy
     leaks) or blocks the call with actionable feedback (for
     structural failures like a missing template section).
- Repo conventions the user *already documented* in
  `.github/PULL_REQUEST_TEMPLATE.md` (or its directory variant) are
  picked up automatically — the user does not re-type the template
  into Boss config.
- The engine has full control of the branch name it tells the worker
  to push to (already true; see `expected_branch_name`), and branch
  prefixes are already configurable per product. The execution id
  component is intentionally preserved in the branch name — it lets
  the engine reliably associate a PR branch with its originating
  execution. Editorial rules expose the **branch naming strategy** so
  products can customise the prefix without losing the cold-path
  detector's "unique by construction" invariant.
- Graceful degradation: every editorial rule kind defaults to "off"
  for an unconfigured product, so this design changes no behaviour
  in `mono` until the user opts in.

## Non-goals

- **A general-purpose linter for human-authored PRs.** Editorial
  rules run only against text Boss agents emit. PRs authored by a
  human (or by a non-Boss agent in the same repo) are out of scope —
  this is not a `danger.js` competitor.
- **Stylistic rewrites.** The redactor strips known-toxic identifiers
  and forces template conformance; it does not rewrite the worker's
  prose for tone, voice, "professionalism," or marketing copy. Those
  decisions belong to the model + repo conventions, not a sanitiser.
- **Per-task or per-execution overrides.** Rules live on the product.
  A task that needs different rules is a sign the product split is
  wrong, not a new tunable. Same philosophy as `dispatch_preamble`.
- **Cross-product rule sharing / inheritance.** v1 stores rules as a
  flat per-product blob. Two products that want the same rules
  duplicate them, the same way two products that want the same
  `dispatch_preamble` duplicate it today.
- **Server-side enforcement on GitHub (CODEOWNERS, branch
  protection rules, required checks).** Out of scope. The engine
  shapes what its own workers emit; what GitHub then accepts is
  the org's concern.
- **Encrypting / hiding the rules from the worker.** The worker's
  initial prompt contains the rule text (so the worker can comply on
  the first try), and the worker can read the same `gh pr` commands
  the hook will reject. There is no adversary here; the editorial
  controls protect the user from the worker's default vocabulary,
  not from a malicious worker.
- **A web UI for editing rules.** v1 is a CLI verb; an app surface
  is a follow-up after the schema settles.
- **Retro-editing existing PR bodies on rule changes.** Rules apply
  at write-time. Past PRs the worker already published stay as they
  were; if the user wants them rewritten they edit by hand or run
  `boss admin rewrite-pr-bodies` (not in v1, but the door is open).

## Background — what actually leaks

Walking the existing worker flow makes the leak surfaces concrete and
keeps the design honest about what each mitigation actually covers.

| Surface                               | Authored by | Controlled by today                                   |
|---------------------------------------|-------------|-------------------------------------------------------|
| Branch name (`boss/exec_<id>`)        | Engine       | `expected_branch_name` in `engine/src/completion.rs`  |
| PR title                              | Worker       | Model + CLAUDE.md / AGENTS.md                         |
| PR body                               | Worker       | Model + CLAUDE.md / AGENTS.md                         |
| Commit message subject                | Worker       | Model + CLAUDE.md / AGENTS.md                         |
| Commit message body                   | Worker       | Model                                                 |
| `gh pr comment` body                  | Worker       | Model                                                 |
| `gh pr review --body` body            | Worker       | Model                                                 |
| `gh issue comment` body               | Worker       | Model                                                 |
| Merge-conflict commit / PR comment    | Worker       | Conflict-resolution prompt + model                    |
| Worker's "final response" to engine   | Worker       | Engine's "## Summary / ## Validation" template        |

The first row (branch name) is **already engine-controlled** — the
worker is told the exact bookmark to push to. Anything that's
engine-controlled today is essentially free to retarget under
editorial rules. The remaining rows are model-authored and need
guidance, with the `gh` invocations being the natural enforcement
choke point because they are tool calls Boss already observes
(`pr_url_capture.rs` already wires a PostToolUse hook on `gh pr` Bash
commands, so the plumbing for a PreToolUse hook on the same surface
is a sibling addition, not new infrastructure).

Boss identifier shapes that should be redactable by default (these
are baked-in patterns the redactor knows about without user config):

- Execution ids: `exec_[0-9a-f]{16}_[A-Za-z0-9]+`
- Project ids: `proj_[0-9a-f]{16}_[A-Za-z0-9]+`
- Task ids: `task_[0-9a-f]{16}_[A-Za-z0-9]+`
- Chore / change ids: `chg_[0-9a-f]{32}` (the cube-lease change ids)
- Cube lease uuids: standard `[0-9a-f-]{36}` is too broad to
  blacklist outright; we only strip uuids that appear inline with
  the literal text "lease" / "cube" within ~40 characters
- Branch namespace: `boss/exec_…` substrings in body text
- Free-text giveaways: `"Boss worker"`, `"the engine"`, `"the
  coordinator"`, `"cube workspace"`, `"work item"`, `"execution id"`,
  `"PostToolUse"`, `"PreToolUse"` — only matched as standalone
  phrases (not inside code blocks; see Q3 quoting rules)

These come from grepping the engine's own prompt strings
(`engine/src/runner.rs`, `engine/src/worker_setup.rs`,
`.claude/CLAUDE.md`) for nouns the worker is exposed to. If the
engine adds new id schemes, they get added here at the same time.

## Alternatives considered

Five distinct shapes were on the table. Two are recommended in
combination; the other three are documented to explain why they
were not picked.

### (A) Prompt-injection only — extend `dispatch_preamble`

Add the editorial rules as another block prepended to the worker's
initial prompt, beneath the existing `[product-preamble]` block:

```text
[editorial-rules]
- Do not mention "Boss", "execution id", or any `exec_…` or `task_…`
  identifier in PR bodies or comments.
- PR body must follow .github/PULL_REQUEST_TEMPLATE.md.
- Do not write a "merge-conflict resolution" comment when fixing
  a conflict in this repo.
[/editorial-rules]
```

**Pros.** Zero new infrastructure beyond a new `products.editorial_rules`
column and rendering the block in `runner.rs::compose_execution_prompt`.
Worker self-corrects most of the time. Cheap to ship.

**Cons.** Model compliance is best-effort. A model that's tired,
distracted by a hard task, or running on a smaller variant will
forget the rule and emit a non-compliant PR. There's no enforcement
— a "do not mention Boss" rule still gets violated, and the human
reviewer notices on GitHub before the engine does. The design's
*goal* is to give the user something they can trust in a work
environment; "the model will probably do the right thing" is not
that.

### (B) PostToolUse sanitiser — rewrite after the fact

After the worker runs `gh pr create --body "<non-compliant text>"`,
the engine's PostToolUse hook fires, the engine reads the resulting
PR, sees the violation, calls `gh pr edit --body "<rewritten>"` to
patch it.

**Pros.** Deterministic; the bad text is fixed without involving the
worker. Compatible with existing `pr_url_capture.rs` flow (same
PostToolUse event).

**Cons.** **Race-y**: the original body lands on GitHub first and
notifications fire ("New PR opened by …" with the bad title in the
subject line) before the rewrite. Anyone subscribed to the repo
sees the leak. PR-edit also re-fires GitHub's webhook chain, which
some CI / review-bot wiring is sensitive to. Most damning: the
worker never learns its body was wrong, so the next PR it opens
has the same problem. (B) treats the worker as an opaque text
generator and fixes its output forever — that's the wrong end of
the loop.

### (C) gh-proxy sub-agent — route every gh call through an editor

Forbid the worker from running `gh pr|issue {create,edit,comment}`
directly. Provide a `boss gh` shim that the worker is told to use
instead. The shim runs the proposed body past a dedicated editorial
sub-agent (a small Claude call with the rules in its system prompt),
which rewrites the body and then performs the real `gh` call.

**Pros.** The editorial agent can apply structural changes the
deterministic redactor can't (template conformance, tone matching).
Worker is shielded from having to understand the rules in detail.

**Cons.** Adds a per-`gh-call` LLM round-trip — latency, cost, and a
new failure mode (the editorial agent is down, the worker's `gh pr
create` is blocked). New tool the worker has to learn — diverges
from the `gh` muscle memory the rest of the Boss prompt builds on.
The sub-agent's rewrites are non-deterministic, so a re-run of the
same `gh pr create` can produce a different body, which makes
testing miserable. The deterministic + advisory split in the
recommended approach gets most of the value with less moving stuff.

### (D) PreToolUse gate — deny non-compliant `gh` calls

The boss-event shim (already wired for every hook event in the
worker's `settings.json`; see `worker_setup.rs:251`) gains a
PreToolUse handler that watches `Bash` tool invocations whose
command matches `gh pr|issue {create,edit,comment,review}`. The
handler extracts the `--body` / `--body-file` / `--title` arguments,
runs them through the deterministic redactor + template-conformance
check, and returns a `permissionDecision` of `allow` (possibly with
a rewritten body) or `deny` with a `decisionReason` the worker reads
and acts on.

**Pros.** The bad body **never reaches GitHub**, so no race with
notifications. The worker is told *why* its call was denied and can
fix the body and retry on the same turn — fast feedback loop. The
plumbing is a sibling to existing `pr_url_capture.rs`'s PostToolUse
handler on the same Bash invocations, so the bash-cmd parsing /
arg-extraction code can be shared. Deterministic: re-runs produce
the same allow/deny decision.

**Cons.** Worker has to handle a denial, which means the worker's
prompt has to tell it editorial rules exist and that denied `gh`
calls are recoverable (this is the `(D)` adjacent to `(A)` story).
Auto-rewrite is conservative — anything the redactor doesn't have
a confident replacement for becomes a denial, which costs a worker
turn. That's an acceptable cost compared to (B)'s "ship-then-edit".

### (E) Branch-name strategy only

Don't touch PR bodies at all. Expose a per-product branch-naming
strategy so teams that want a different prefix can set one.

**Pros.** One engine-side change with no worker-facing protocol. The
execution id component of the branch name is retained (it gives the
engine a reliable PR-to-execution association); only the prefix is
customisable.

**Cons.** Does nothing for the PR body / comments, which is where
most internal taxonomy leakage occurs. Worth doing as part of the
recommended approach, but not sufficient on its own.

### Recommendation: (A) + (D) + (E), staged

A combination ships, in this order so each phase delivers value on
its own and the later phases build on the earlier ones:

1. **(E)** — per-product branch-naming strategy. Smallest change,
   no worker-facing protocol. Products that want a different prefix
   get it immediately; the execution id remains in the branch name
   so the engine can associate PR branches with their originating
   execution.
2. **(A)** — `products.editorial_rules` column + `[editorial-rules]`
   block injected into the worker prompt + a baked-in default rule
   set that redacts Boss identifier shapes. Most PRs comply on the
   first try after this.
3. **(D)** — PreToolUse hook on `gh pr|issue {create,edit,comment,
   review}` Bash commands, applies the same rules deterministically,
   either auto-rewrites (regex-replaceable leaks) or denies with
   actionable feedback (structural failures, template missing
   sections, mentions of forbidden phrases the redactor isn't
   confident enough to silently strip).

(B) and (C) are explicitly rejected; the design notes them so future
"what about a sanitiser" / "what about a sub-agent" suggestions land
on the existing trade-off discussion rather than re-litigating it.

## Chosen approach

### Scope: per-product

The rule record lives on the `products` row. One product owns one
repo's worth of work (current invariant) so per-product == per-repo
for the foreseeable future. When multi-repo modelling lands
(`proj_18a2bbe20fc03718_8`), the per-repo dimension can be promoted
without disturbing the rule shape — a separate-repo workspace inherits
the product's rules until proven otherwise.

The natural alternative (per-repo, keyed by `repo_remote_url`) loses
the user's ability to say "this product's work uses these rules but
that product's work uses different rules even though they share a
repo" — which today is hypothetical, but the per-product scope keeps
that door open at no extra cost.

### Rule shape: structured, not free-form

Editorial rules are a small JSON blob on the product row:

```rust
pub struct EditorialRules {
    /// Free-form markdown injected into the worker's initial prompt
    /// under `[editorial-rules]`. The user's primary tool for
    /// expressing repo conventions that don't fit a structured field
    /// ("don't mention competitor names", "match the existing PR
    /// title style: <Component>: <imperative verb> <noun>").
    pub instructions: Option<String>,

    /// Additional redaction patterns layered on top of the baked-in
    /// Boss-identifier list. Each pattern is a regex; the replacement
    /// is either a literal string ("[redacted]" is a common choice)
    /// or the empty string (strip and collapse whitespace).
    pub redactions: Vec<RedactionRule>,

    /// Template policy. The hook checks PR bodies against the repo's
    /// `.github/PULL_REQUEST_TEMPLATE.md` (or
    /// `.github/PULL_REQUEST_TEMPLATE/<name>.md` for the directory
    /// form) at the level set here.
    pub template_policy: TemplatePolicy,

    /// Naming strategy for the bookmark the engine tells the worker
    /// to push to. The default `BossExecPrefix` preserves today's
    /// behaviour; `OpaqueHash` swaps to a content-hash form that
    /// doesn't leak the literal execution id.
    pub branch_naming: BranchNaming,

    /// Optional override for the engine-injected commit-message
    /// trailer (today's `Co-Authored-By: Claude …` line) — workers
    /// don't currently emit one, but `claude` may start to, and the
    /// rule lets the user say "no, our org policy is no AI trailer."
    pub commit_trailer_policy: TrailerPolicy,
}

pub struct RedactionRule {
    /// Anchored regex. Matched against the candidate body text after
    /// quote-stripping (see Q3).
    pub pattern: String,
    /// Literal replacement. Empty string means strip + collapse.
    pub replacement: String,
    /// Whether the hook applies this as an auto-rewrite (`Rewrite`)
    /// or treats it as a denial signal (`Block`). `Rewrite` is the
    /// default for short patterns with confident replacements;
    /// `Block` is the right choice when stripping would change the
    /// meaning of the body.
    pub kind: RedactionKind,
}

pub enum RedactionKind { Rewrite, Block }

pub enum TemplatePolicy {
    /// No template check. The default for personal products.
    Off,
    /// Soft check: warn in the prompt if the repo has a template the
    /// worker didn't follow, but don't block at the hook.
    Advise,
    /// Hard check: at write time, the hook compares the PR body's
    /// section headings against the template's section headings and
    /// blocks if mandatory sections are missing or blank.
    Enforce,
}

pub enum BranchNaming {
    /// `boss/{execution_id}` — today's behaviour. The execution id
    /// lets the engine reliably associate the branch with its originating
    /// execution; the `boss/` prefix is configurable if the repo's
    /// naming conventions require something different.
    BossExecPrefix,
    /// `boss/{first_8_chars_of_sha256(execution_id)}` — shorter form,
    /// still unique-by-construction per execution.
    OpaqueHash,
    /// `{user_prefix}/{first_8_chars_of_sha256(execution_id)}` —
    /// the user supplies a custom prefix; uniqueness is preserved by
    /// the content-hash suffix.
    CustomPrefix { prefix: String },
}

pub enum TrailerPolicy {
    /// Worker writes commit messages as-is; whatever trailer `claude`
    /// adds, ships.
    Default,
    /// The hook strips trailing `Co-Authored-By:` lines from commit
    /// message bodies on `git commit -m` / `jj describe -m`
    /// invocations. (v1 is allowed to no-op this; see Risks.)
    NoAiTrailer,
}
```

JSON column on `products`. Three nullable fields on `Project` was the
right call for the design-doc pointer (filterable, indexable), but
editorial rules are read-by-id and never filtered, so a JSON blob is
the right shape — same shape decision as `external_tracker_config`
(`Product.external_tracker_config: Option<serde_json::Value>`,
`products` row).

### Wire-up at three engine points

#### Point 1 — Branch naming, at `completion::expected_branch_name`

`expected_branch_name(execution_id)` becomes
`expected_branch_name(execution_id, branch_naming)`. The cold-path
detector (`detect_pr` callers) takes the same arg pair so it
reconstructs the same name from `state.db` alone. The `BranchNaming`
strategy is read off the product's `editorial_rules` at execution
spawn time and stored on the `executions` row (so a rule change after
spawn doesn't break the detector's reconstruction — the row remembers
which scheme it was opened under). This mirrors how
`expected_branch_name` is *already* read off `executions.id` rather
than recomputed from the chore row; the same "snapshot at spawn,
detect deterministically forever after" invariant holds.

`OpaqueHash` uses `&hex::encode(Sha256::digest(execution_id))[..8]`,
which is collision-resistant within an org's execution-id space (the
execution id is itself unique; an 8-hex prefix gives 32 bits of
collision space, sufficient for ~65k executions before any
birthday-paradox concern). If two products both pick `OpaqueHash`
and one's hash collides with another's open PR, `gh pr list --head
<hash>` still resolves uniquely because the lookup is scoped per
`repo_remote_url`. Cross-repo collisions are not collisions.

#### Point 2 — Prompt injection, at `runner::compose_execution_prompt`

Today the prompt assembly in `engine/src/runner.rs` already prepends
`[product-preamble]\n{}\n[/product-preamble]` when
`product.dispatch_preamble` is set. The same pattern adds an
`[editorial-rules]` block when `product.editorial_rules.instructions`
is non-empty or the baked-in defaults are active. The block is
rendered as:

```text
[editorial-rules]
**Editorial rules for PRs / GitHub comments in this product.**
Apply these rules to every PR title, PR body, PR / issue comment,
commit-message body, and merge-conflict note you write for this run.

Baked-in rules (always apply):
- Do not mention Boss execution / project / task / chore identifiers
  in user-facing text. The shapes are `exec_…`, `proj_…`, `task_…`,
  `chg_…`. They are internal vocabulary that humans on this repo
  have no context for.
- Do not refer to "Boss worker", "the engine", "the coordinator",
  "cube workspace", or "work item" in user-facing text — these are
  internal Boss vocabulary.
- When referring to your branch in PR text, say "this branch" rather
  than its full name — the branch name is engine bookkeeping (it
  associates the PR with its originating execution) and is not
  meaningful to human reviewers.

Product-specific rules (configured on this product):
{user-supplied instructions verbatim}

Template policy: {Off | Advise: see <path> | Enforce: see <path>}
- If Advise / Enforce, the PR body must follow the structure of
  `.github/PULL_REQUEST_TEMPLATE.md` (rendered below).

```PR template```
{verbatim contents of the repo's template, if one exists and policy != Off}
```

Enforcement:
The engine's PreToolUse hook intercepts `gh pr create`, `gh pr edit`,
`gh pr comment`, `gh pr review`, and `gh issue comment` invocations.
If your body / title violates a rule, the call is denied or
rewritten and you will see feedback. Comply on the first try when
you can — denials cost a worker turn.
[/editorial-rules]
```

The block is injected *after* the workspace-rules / RESUME-EXISTING-PR
preamble and *before* the per-kind directive (the design or
implementation block), so the rules are top-of-mind without
overriding the higher-priority "this run has an open PR, push to it"
directive.

For products with `editorial_rules` left at its default (which is
"baked-in only, no instructions, template_policy: Off, branch_naming:
BossExecPrefix"), the block is rendered with just the baked-in
identifier-redaction rules and *no* template / instructions section.
Personal products see a one-paragraph block that costs ~80 tokens
and makes a meaningful dent in identifier leakage even when the
user hasn't configured anything.

#### Point 3 — PreToolUse hook on `gh` invocations

The boss-event shim already forwards every PreToolUse event to the
engine. The engine grows a new handler for the subset of Bash
invocations matching:

```text
^\s*(GIT_DIR=\S+\s+)?gh\s+(pr|issue)\s+(create|edit|comment|review)\b
```

(the same envelope `pr_url_capture::is_gh_pr_invocation` already
recognises; the regex moves to a shared module). The handler:

1. Parses out `--body`, `--body-file` (reads the file), `--title`,
   `--message`, `--editor`, and `--web` flags.
2. If `--editor` or `--web` is present, the hook *allows* the call
   (the worker is delegating to the human / browser — no body the
   hook can inspect).
3. Otherwise: runs the body / title through the redactor, applies
   `Rewrite`-kind redactions in place, collects `Block`-kind hits.
4. If `template_policy == Enforce` and the call is `gh pr create`
   or `gh pr edit`, compares the body's headings to the template's
   required headings, collecting any missing sections.
5. Decides:
   - **No findings** → return `{ "permissionDecision": "allow" }`.
   - **All findings are auto-rewritable, and the rewritten body
     differs from the worker's original** → rewrite the Bash command
     to use the new body (substitute the `--body` arg's value, or
     overwrite the `--body-file`'s contents) and return
     `{ "permissionDecision": "allow", "decisionReason": "Editorial
     hook rewrote: <list>" }`. The worker sees the success path with
     a one-line explanation it can echo in its final summary if it
     wants.
   - **Any finding is `Block` or a template / structural failure**
     → return `{ "permissionDecision": "deny", "decisionReason":
     "Body violates editorial rules: <numbered list>; please fix
     and retry." }`. The worker reads the deny reason, edits the
     body, and re-issues `gh pr create`.

Step 3's "in-place rewrite" is the only delicate part: claude's
PreToolUse contract supports `permissionDecisionInput` mutation,
which lets the hook return a *modified* `tool_input` instead of the
original. The hook substitutes the rewritten body into the
`tool_input.command` string (the same parsing the redactor already
did), and claude runs the modified command.

For `--body-file <path>`, the hook *overwrites* the file at the
referenced path on disk before allowing the call. The path is
inside the leased workspace (the worker is the only writer), so
there's no shared-state concern.

#### Failure handling on the hook

- **The engine is unreachable** (boss-event buffers the event for
  later delivery): the PreToolUse hook returns nothing within its
  bounded retry window, claude's default is `allow`, the worker
  pushes potentially-non-compliant text. Same failure-open posture
  as every other PostToolUse the engine has — the design treats
  editorial controls as advisory in a partition rather than
  blocking the worker mid-task. Cleanup is a follow-up `boss admin
  audit-pr-bodies <product>` chore that runs the rules over the
  product's open PRs and emits attention items for any violations.
- **The redactor's regex is too greedy and strips legitimate text**:
  the worker sees the rewrite in `decisionReason`, can spot the
  over-rewrite, and either edits the body to restore the lost text
  in a follow-up `gh pr edit` (which goes through the hook again,
  rinse + repeat) or stops with a human-visible "the editorial rule
  X is too aggressive for this PR" note in its final summary. The
  user adjusts the regex.
- **The body is fundamentally non-compliant** (e.g. the worker
  insists on using a forbidden phrase because the user asked the
  product to ship something the rules say not to mention): the hook
  denies, the worker retries, the hook denies again. After three
  denies on the same `gh pr create` invocation within one execution
  (the engine tracks this in-memory), the hook flips to *allow with
  a `WorkAttentionItem`* — the body ships, but the user gets a
  flagged item to review. Prevents infinite loops without
  silently surrendering the rule.

### Default rules — every product gets the baked-ins

Every product gets the identifier-redaction list and the
"don't say Boss / engine / coordinator / work item" plain-text rules
applied *without* any per-product config. This is the design's
ergonomic answer to "I don't want to write rules for my personal
product but I'd still like my PRs to not say `exec_18b07a…`":

- Baked-in identifier redactions: the regex list under "what leaks"
  above, applied as `Rewrite` (replacement = empty string +
  whitespace collapse).
- Baked-in phrase redactions: `Boss worker`, `the engine`, `the
  coordinator`, `cube lease`, `cube workspace`, `work item`,
  `execution id`. Applied as `Block` (the worker is told what to
  rephrase rather than silently scrubbed, because phrase deletion
  is more likely to mangle a sentence than id-deletion is).
- Default `template_policy = Off`, default `branch_naming =
  BossExecPrefix`, default `commit_trailer_policy = Default`. Each
  one of those defaults preserves today's behaviour exactly. The
  user opts in to `Advise` / `Enforce` / `OpaqueHash` / `NoAiTrailer`
  per product.

The intent: shipping this feature does not require the user to
configure anything for their personal projects to benefit, and a
work product gets meaningful editorial controls by setting
`template_policy = Enforce` + `branch_naming = OpaqueHash` + a
short `instructions` block.

### Surfacing what the hook did

Every hook decision (allow / allow-with-rewrite / deny) is forwarded
to the engine over the events socket as a new
`work_editorial_action` event. The engine writes it to a per-product
audit table (`editorial_actions`) so the user can answer "which of
my PRs had bodies rewritten by the hook?" with a CLI:

```sh
boss editorial show <selector>            # last 50 actions on this product
boss editorial show <selector> --pr 1234  # actions on this PR
```

This matters in a work environment where the user wants to know that
the hook is actually doing things (and isn't silently no-opping). It
also makes the rule-tuning loop concrete: see what got blocked, decide
if the rule was right, adjust.

`boss editorial test` is the rule-author's REPL — they edit

### Engine module split

- `engine/src/editorial.rs` (new) — rule loading, baked-in default
  list, regex compilation + caching, `evaluate(body, rules) ->
  EditorialDecision` (the pure function).
- `engine/src/runner.rs` — assemble the `[editorial-rules]` block
  in `compose_execution_prompt`, read `editorial_rules` off the
  product row.
- `engine/src/completion.rs` —
  `expected_branch_name(execution_id, branch_naming)` overload;
  callers in `pr_url_capture` and `conflict_watch` thread the new
  arg through.
- `engine/src/pr_url_capture.rs` — refactor `is_gh_pr_invocation`
  out to a shared `gh_invocation::classify` so the editorial
  PreToolUse handler reuses it.
- `engine/src/app.rs` (or wherever PreToolUse dispatch lives) — new
  handler `handle_pretooluse_for_editorial` that calls
  `editorial::evaluate` and serialises the decision to the hook's
  expected JSON.
- `cli/src/commands/product.rs` — `set-editorial-rules` verb.
- `cli/src/commands/editorial.rs` (new) — `show` / `test` verbs.

### Topic / event

`work_editorial_action` is a new event, forwarded over the existing
events socket. The kanban can subscribe and badge a product card
when actions accumulate; v1 doesn't ship that UI.

## Risks / open questions

**R1 — Hook latency on the worker's hot path.** PreToolUse handlers
run synchronously between claude deciding to run `gh pr create` and
the bash invocation firing. If the editorial evaluator takes more
than ~100ms, the worker feels slow. Mitigation: the evaluator is
pure regex over a body (typically < 5KB); benchmark target is < 5ms
per call. Pre-compile regexes at engine startup and cache by product.

**R2 — Auto-rewrite can corrupt code blocks.** A worker that wrote
`This fixes the bug where exec_18b07a506d2518d0_1b appears in
output` is fine to rewrite. A worker that wrote a *literal example*
of a Boss id inside a fenced code block (`This system uses ids like
\`exec_…\``) is not, but the redactor will strip it anyway.
Mitigation: the evaluator parses the body's markdown structure
(commonmark; the repo already depends on a markdown parser for the
designs renderer) and skips matches inside fenced code blocks. Inline
code spans are matched but with a higher confidence bar (only
matches anchored as full inline-code-span contents trigger).

**R3 — Worker oscillation on repeated denies.** The "three denies
then allow with attention item" mitigation prevents the worker from
infinite-looping on a body it can't make compliant, but the user
sees a non-compliant PR with an attention item rather than a
blocked one. Open question: is this the right trade-off, or should
the engine harder-stop and ask the user? Recommend the soft fail
for v1 — a half-shipped PR with a flag is more recoverable than a
worker stuck mid-execution with no PR at all.

**R4 — Template-conformance check is brittle to template format.**
GitHub's PR-template syntax is just markdown with HTML comments for
placeholder hints. Some templates use H2 headings as section
markers, some use bold text, some use checkboxes. The "missing
section" detector has to make conservative assumptions about what's
a section. Mitigation: v1 only enforces H2 / H3 headings — if a
required heading is missing from the body, the hook blocks. Other
formats degrade to `Advise` (warn in the prompt, don't block).
Open question: do we ship a `template_format = strict_h2 | loose`
knob in v1 or wait for a user to complain? Recommend wait.

**R5 — Branch-name strategy change after a worker has already
opened a PR.** If the user flips `branch_naming` between executions
on the same project, the next execution's branch follows the new
strategy. PRs already open under the old strategy don't move. The
detector reconstructs the branch from `executions.branch_naming`
(snapshot at spawn), so the cold path stays correct. Documented;
no remediation needed.

**R6 — `OpaqueHash` collisions across products.** Two executions
in two different products can produce the same 8-hex prefix. As
long as the cold-path detector scopes its `gh pr list --head`
query by `repo_remote_url`, this is fine. Risk: if a future
detector ever globs across products (it shouldn't, but), the
collision becomes a real fan-out vector — same shape as incident
001, different root cause. Mitigation: assert in the detector that
the query is scoped, and add a test that two products with the
same hash see distinct PRs.

**R7 — Worker can bypass the hook by writing to GitHub via curl
or a non-`gh` tool.** A worker that's hostile to the editorial
rules could `curl -X POST https://api.github.com/repos/.../pulls`
directly and bypass the hook. Two responses:
1. Boss workers are not adversaries; the rules protect the user
   from the model's defaults, not from a malicious agent.
2. If we ever need adversarial-grade enforcement, it has to move
   off the worker entirely (option (C) above), and that's a
   different design. v1 documents the limitation.

**R8 — `--body-file` overwrite races.** The hook writes the
rewritten body back to the `--body-file` path before allowing the
call. If the worker spawns a parallel process that's also reading
the file, the parallel reader could see the rewritten contents.
Mitigation: workers do not spawn parallel readers of body files;
the file is written by the worker, the hook rewrites it, the
`gh` invocation reads it, and nobody else touches it. No
mitigation needed; documented.

**R9 — Commit-trailer enforcement (`NoAiTrailer`) is partial.**
The hook can match `git commit -m '…'` (claude's typical commit
form), but a `git commit` without `-m` opens `$EDITOR` and the
hook can't see the body. Per the existing worker setup,
`$EDITOR` is intentionally unusable in the worker (see CLAUDE.md
"Commit messages must be inline"), so the rule covers the common
case. Mitigation: document the gap; the user is informed that
`NoAiTrailer` is best-effort.

**R10 — Editorial rules drift from baked-in identifier patterns.**
If the engine introduces a new id format (`run_…`, say) and the
baked-in regex list isn't updated, the new id leaks. Mitigation:
the id-format constants in `engine/src/work.rs` (or wherever they
live) get the editorial-redactor as a downstream consumer in a
shared test — adding a new id family fails the test until the
redactor learns the new pattern.

**R11 — Coordination with `dispatch_preamble`.** Both
`[product-preamble]` and `[editorial-rules]` are injected into the
worker's initial prompt. They are sibling blocks, not nested —
the rendering order is `[product-preamble]` first (matches today),
then `[editorial-rules]`. A user who has put editorial-style
content into `dispatch_preamble` today migrates by moving it into
`editorial_rules.instructions`; the engine doesn't auto-migrate.
The `boss product show` line can flag products that have both,
suggesting consolidation.

**R12 — Where do rules apply to the worker's *final response*?**
The worker's final response to the engine (the `## Summary / ##
Validation / ## Open Questions` block) is not GitHub-visible but
is shown to the user and stored in the engine's transcript. The
recommendation is to **not** apply editorial rules to the final
response — it's an internal artefact and stripping identifiers
from it makes debugging harder. Document this asymmetry; if a
user wants the final response sanitised too, they say so and we
add a `apply_to_worker_summary: bool` flag.

**R13 — Template policy interaction with the engine's own
"Respond with `## Summary / ## Validation / ## Open Questions`"
instruction in `runner.rs`.** That sectioning is for the worker's
*final response*, not the PR body. Workers historically use the
same sectioning for their PR body (because it's what they just
wrote). With `template_policy = Enforce` + a repo template that
uses different headings, the worker has two competing structures
and may produce a PR body with both, then `## Summary` etc. at
the bottom. Mitigation: the `[editorial-rules]` block in the
prompt is explicit that "the PR body must match the template
*regardless of the final-response sectioning rules*." Test
empirically; if compliance is poor, revisit.

**R14 — `--body-file` paths can be relative.** The hook resolves
relative paths against the worker's cwd (the leased workspace
root). If the worker has changed cwd, the rewrite hits the wrong
file or no file. Mitigation: the PreToolUse payload carries the
worker's cwd (claude's PreToolUse hook payload includes
`tool_input.cwd` when present); the hook uses that.

## Follow-up implementation chores (to enqueue once approved)

Each fits one worker session.

1. **Schema + migration.** `products.editorial_rules` JSON,
   `executions.branch_naming` text, `editorial_actions` table.
   Idempotent. Existing rows default to NULL.
2. **Protocol types.** `EditorialRules`, `RedactionRule`,
   `RedactionKind`, `TemplatePolicy`, `BranchNaming`,
   `TrailerPolicy`, `EditorialAction`. Mirror in `Models.swift`.
3. **`engine/src/editorial.rs` — pure evaluator.** `evaluate(body,
   title, rules) -> EditorialDecision`. Baked-in default list. Unit
   tests for every redaction kind, code-fence skipping, template
   missing-section detection.
4. **Engine: `expected_branch_name(execution_id, branch_naming)`
   overload + plumb the new arg through `pr_url_capture` and
   `conflict_watch`.** Acceptance: existing tests pass with
   `BossExecPrefix`; new tests cover `OpaqueHash` and
   `CustomPrefix`; detector reconstructs the right name from
   `executions.branch_naming`.
5. **Engine: `compose_execution_prompt` adds `[editorial-rules]`
   block.** Acceptance: default-config product renders the
   baked-in identifier rules only; configured product renders
   the user instructions + template + enforcement banner.
6. **Engine: PreToolUse handler for `gh pr|issue {create,edit,
   comment,review}` Bash invocations.** Acceptance: integration
   test against a stub `gh` shim — a worker that runs
   `gh pr create --body "<bad body>"` sees a deny; a worker that
   runs with a body containing redactable identifiers sees an
   allow-with-rewrite and the substituted body lands.
7. **Engine: `editorial_actions` event emit + table writes.**
   Acceptance: action rows appear after every hook decision; CLI
   `boss editorial show` lists them.
8. **CLI: `boss product set-editorial-rules` /
   `boss editorial show` / `boss editorial test`.** Acceptance:
   `--help` for each verb; integration test for the round-trip.
9. **PR-template loader.** Read `.github/PULL_REQUEST_TEMPLATE.md`
   and `.github/PULL_REQUEST_TEMPLATE/*.md` from the leased
   workspace at spawn time; cache per product per workspace lease.
   Acceptance: covers single-file, directory-form, missing-file.
10. **(Optional follow-up) `boss admin audit-pr-bodies <product>`.**
    Scan open PRs against the rules and emit a `WorkAttentionItem`
    per violation. Out of v1; documented as the cleanup path for
    when the hook fails open (R-engine-unreachable).
11. **(Optional follow-up) Sub-agent rewriter for `Advise`-tier
    template policy.** When the deterministic redactor can't fix a
    structural issue but the user wants the PR shipped anyway, a
    one-shot Claude call rewrites the body to fit the template.
    Strictly opt-in; reuses the project's design-renderer infra.

## Out of scope

- Editing existing PRs on rule change (admin chore covers this if
  ever needed).
- General-purpose linting of human-authored PRs.
- Stylistic / tone rewrites — the rules are about identifier leakage
  and template conformance, not prose quality.
- Per-execution overrides — rules live on the product.
- A web UI for editing rules.
- Adversarial enforcement (a worker that wants to bypass the rules
  can always use `curl`; out of scope per R7).
- Encrypting rules at rest — the worker reads them in plaintext on
  spawn anyway.
